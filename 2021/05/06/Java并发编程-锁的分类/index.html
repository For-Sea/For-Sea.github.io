Java并发编程-锁的分类

发表于 2021-05-06 分类于 Java并发编程

Java并发编程中，锁的分类与适用场景

锁的分类

  • 互斥锁
  • 自旋锁
  • 读写锁
  • 悲观锁
  • 乐观锁

互斥锁与自旋锁

互斥锁与自旋锁是最基础的两类锁，很多高级的锁是基于这二者实现的，所所以我们有必要了解二者的区别。

我们知道，线程加锁的目的是保证在任意时间内，共享资源只被一个线程访问，以此避免多线程访问导致共享数据错乱。所以，当有一个线程获取共享资源的锁之后，其他线程对共享资源加锁就会失败。

而互斥锁与自旋锁的区别在于线程加锁失败后的处理方式：

  • 互斥锁：加锁失败后，线程释放CPU，进入阻塞队列，等待资源被释放。
  • 自旋锁：加锁失败后，线程进入忙等待状态，直到其拿到锁。

简单来说，当线程加锁失败时，互斥锁使用“线程切换”来应对，自旋锁使用“忙等待”来应对。

互斥锁的线程切换需要从用户态进入到内核态，由操作系统完成，因此开销成本比较大；自旋锁在用户态进行忙等待，直到锁空闲，然后获取锁，因此不需要进入内核态，开销较小。

因此，如果代码执行时间很短，应该使用自旋锁（自旋锁等待时间短，否则互斥锁光是进行线程切换所需的时间就比代码执行时间长了），否则使用互斥锁。

互斥锁和自旋锁是其他高级锁的基础，高级锁可以选择其中的一种进行实现，比如读写锁可以使用互斥锁实现，也可以使用自旋锁实现。

读写锁的读与写

读写锁分为读锁和写锁，读取共享资源时使用读锁，修改共享资源时使用写锁，读写锁适用于区分读写操作的场景。

读锁是共享锁，可以被多个线程共享，原因是读操作不会修改共享资源，而多个线程并发执行读操作可以提高效率。

写锁是独占锁，任一时刻只能由一个写锁独占共享资源，其他读锁跟写锁将会阻塞，原因是写操作会修改共享资源。

综上，当读写锁在读多写少的场景时，能发挥出优势。

读写锁还可以细分为读优先锁跟写优先锁。

读优先锁：当线程A持有读锁时，线程B请求写锁，会被阻塞，此时后续线程C请求读锁，线程C仍然可以获得读锁；只有在线程A、C释放完读锁后，线程B才能成功获取写锁。

写优先锁：当线程A持有读锁时，线程B请求写锁，会被阻塞，此时后续线程C请求读锁，也会被阻塞；线程A释放读锁后，线程B获得写锁，线程B释放写锁后，线程C获得读锁。

读优先锁和写优先锁都会造成饥饿现象，读优先锁可能会让请求写锁的线程进入饥饿，写优先锁可能会让请求读锁的线程进入饥饿，因此为了避免这两种情况，可以使用“公平读写锁”，即不偏袒任何一方，只是将请求锁的线程放在阻塞队列，然后根据先进先出的原则依次获得锁。

读写锁可以根据场景选择互斥锁跟自旋锁中的一种来实现。

悲观锁与乐观锁

悲观锁认为共享资源会被修改的概率很大，所以每次修改共享资源时都会上锁，防止其他线程修改，互斥锁、自旋锁、读写锁都是悲观锁。乐观锁认为共享资源会被修改的概率很小，所以不加锁，也叫无锁编程，但是乐观锁采用了CAS方式来保证共享资源的修改不会产生冲突。

采用乐观锁的工作方式时，线程假定共享资源会被其他线程修改的概率很低，于是线程会直接修改共享资源，然后验证在修改期间有没有其他线程完成了修改，如果没有，则直接完成修改，如果有其他线程完成了修改，则放弃本次操作，之后再重试。注意，乐观锁一旦发生冲突，重试的成本很高。

悲观锁适用于冲突概率非常较高，且加锁成本较低的场景。

乐观锁适用于冲突概率非常低，且加锁成本非常高的场景。

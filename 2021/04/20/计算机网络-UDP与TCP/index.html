计算机网络-UDP与TCP

发表于 2021-04-20 分类于 计算机网络

UDP？TCP？为什么要两个传输协议？

什么是UDP？

即用户数据报协议（User Datagram Protocol），是运输层两种传输协议的一种，负责传输用户数据报。

UDP的特点和适合场景

UDP是一种无连接、不可靠的传输协议，其特点包括：

  • 无连接
  • 提供尽最大努力交付，即不保证可靠交付
  • 面向报文
  • 没有拥塞控制
  • 支持一对一、一对多、多对一、多对多的交互通信
  • UDP报文首部开销小

由于其无连接的特性，UDP显得简单有效，传输数据的效率比TCP快，但由于其不可靠的特性，不能用于重要文件的传输，所以UDP协议多用于视频聊天、直播等场景。

什么是TCP？

即传输控制协议（Transmission Control Protocol），是运输层的另一个用于传输的协议，传输的是报文段。

TCP的特点和使用场景

TCP是一种面向连接、可靠的传输协议，其特点如下：

  • 面向连接
  • 点对点传输
  • 提供可靠交付
  • 提供全双工通信，即通信双方可以随时向对方发送数据
  • 面向字节流

由于TCP面向连接、可靠的特性，其多用来传输重要文件、大文件。

TCP可靠传输

理想的可靠传输包括：一是传输不产生差错，二是传输的数据能及时被处理。

可靠传输的原理：停止等待协议、ARQ协议。

可靠传输的实现：滑动窗口、流量控制、拥塞控制、连接管理。

什么是停止等待协议？

“停止等待”就是发送方发送完一个分组，停止发送，等待对方确认接收后再发送下一个分组。即先停止，后等待。

发送方发送完分组并等待确认的过程中，可能会出现四种情况：

  1. 无差错：正常情况，发送方发送一个分组，收到确认后发送下一个分组。
  2. 超时重传：发送方发送一个分组，迟迟等不到确认，可能是因为此次发送的报文有差错或者丢失了，此时在超过一定等待时间后，发送方会重新发送分组。
  3. 确认丢失：发送方发送一个分组，迟迟等不到确认，可能是因为接收方发送的确认丢失了，这也是超时重传的一种情况。此时发送方因为超时重传会再发送一次分组，接收方就会收到重复的分组。接收方应该将重复分组丢弃，并再次向发送方发送确认。
  4. 确认迟到：发送方发送一个分组，迟迟等不到确认，可能是因为接收方发送的确认迟到了，这也是超时重传的一种情况。此时发送方会因为超时重传再发送一次分组，然后发送方会收到重复的确认，发送方会直接丢弃重复确认。

什么是ARQ协议？

ARQ协议即自动重传请求（Automatic Repeat reQuest）协议，其实就是超时重传，意思是发送方不需要等待接收方的错误提示信息，而是直接根据情况自动重新发送分组。

什么是连续ARQ协议？

连续ARQ协议是滑动窗口的基础，滑动窗口是提高TCP传输效率的一种实现方式。

连续ARQ协议的基本概念是：

发送方将要发送的分组编上序号，一次发送多个分组，比如分组一次发送分组1-5，则可以认为发送的窗口大小为5。

而接收方发送的是累积确认，即只发送按序到达的最后一个分组的确认，不发送其之前的确认，比如最后一个按序到达的分组为5，则发送分组5的确认，这表明分组5之前的分组都成功接收了。

接收到确认后，发送方的发送窗口会向前移动X个分组（X为成功接收到的分组），比如：如果接收到分组5的确认，说明分组1-5都已成功接收，接下来发送分组6-10；如果接收到分组3的确认，说明分组1-3都已成功接收，接下来发送分组4-8。

发送方发送的过程中，发送的分组数就是滑动窗口的大小，发送的多个分组就是一个滑动窗口。

滑动窗口

前面了解了连续ARQ协议，对滑动窗口有了一定了解，下面来看看滑动窗口。

TCP的滑动窗口以字节为单位，使用滑动窗口实现可靠传输的过程如下：

  1. 假如接收方的接收窗口大小为20个字节，则发送方一开始的发送窗口也为20个字节。

  2. 发送方发送字节31-50，此时有两种情况。

    1. 字节按序到达：比如31到达，接收方会发送32，窗口向后滑动一个字节；32到达，接收方发送33，窗口向后滑动一个字节……以此类推。
    2. 字节没有按序到达：比如33先到达，31、32没有到达，接收方会先把33存放在缓存中，等到31、32到达后，发送34，窗口向后滑动三个字节。

  4. 重复步骤2。

其中需要注意的是

  1. 接收窗口与发送窗口：一开始的发送窗口大小由接收窗口大小决定，发送窗口与接收窗口一样大；但是在传输过程中，由于具有延迟，发送窗口不总是与接收窗口一样大。
  2. 累积确认：接收方发送的是期望接收下一个字节的序号。
  3. 解决无序接收：接收方接收到的字节不一定是有序的，在接收到的字节之前可能还有一些没有到达的字节，此时接收方会将这些无序到达的字节放在缓存中，等待前面的字节到达后再组成有序的字节，并向发送方发送想接收的下一个字节序号。

这里关于滑动窗口讲的很粗糙，建议再去查阅其他资料。

拥塞控制

什么是拥塞控制？

尽可能避免网络通讯的拥塞，防止传输过程效率变低甚至死锁。

发送方的发送窗口大小由接收窗口（rwnd）和拥塞窗口（cwnd）决定，其中接收窗口大小是接收方告诉发送方的，而拥塞窗口大小是发送方根据网络拥塞情况自己设置的。发送窗口=Min（接收窗口，拥塞窗口）。

慢开始和拥塞避免：先假设网络是拥塞的，所以开始时cwnd为1，但之后随二次方指数增长；当到达慢开始门限值ssthresh后执行拥塞避免，此时cwnd线性增长，当发生超时重传后再次进入到慢开始，但这次将慢开始门限值ssthresh设为原来的一半。

快重传与快恢复：快重传即当收到3次重复的确认后，马上发送接收方要求的报文段；快恢复即发生拥塞时，从发生拥塞时的cwnd大小的一半开始执行拥塞避免，并将ssthresh设置为发生拥塞时cwnd大小的一半。

TCP连接控制

创建TCP连接-三次握手

什么是三次握手？

在发送和接收双方之间建立TCP连接时，需要进行三次单方向传输：发送方–>接收方、接收方–>发送方、发送方–>接收方。如图：

三次握手的目的是什么？

使发送和接收双方都能知道对方的存在以及对方的收发功能没有问题。

第一次握手：接收方知道自己的接收功能、对方的发送功能没问题，发送方什么都不知道；

第二次握手：接收方知道自己的接收功能、对方的发送功能没问题，发送方知道自己的发送功能、自己的接收功能、对方的接收功能、对方的发送功能没问题；

第三次握手：接收方知道自己的发送功能、自己的接收功能、对方的发送功能、对方的接收功能没问题，发送方知道自己的发送功能、自己的接收功能、对方的接收功能、对方的发送功能没问题。

两次握手行不行？

试想两次握手时发生以下两种情况：

  1. 发送方发送TCP连接请求，但是请求丢失了。第一次发送的TCP连接请求丢失，于是发送方第二次发送连接请求，接收方收到连接请求发送确认，发送方接收到确认后直接建立TCP连接。看起来好像没有什么问题，但是发生下一种情况时就有问题了。
  2. 发送方发送TCP连接请求，但是请求迟到了。第一次发送的TCP连接请求迟到了，发送方超时重传第二次TCP连接请求，然后接收方接收到请求后发送确认，发送方收到确认后建立连接并传输数据。当数据传输完成并断开连接后，此时第一次发送的TCP连接请求才到达，于是双方再次建立了连接，但是要传输的数据已经传输完了，所以双方不会传输数据，只会维持这个TCP连接，白白浪费资源。

如果是三次握手，当迟到的第一次发送的TCP连接请求到达后，接收方发送确认，发送方接收到确认后不会发起第三次握手，则连接不会建立。

断开TCP连接-四次挥手

什么是四次挥手？

在断开TCP连接时，需要进行四次单方向传输。

四次挥手的目的是什么？

当发送方发送TCP关闭连接请求后，接收方响应确认，此时发送方就不再向接收方发送数据了，整个TCP连接处于“半关闭”状态；此时接收方如果有想向发送方传输的数据可以继续传输，传输完毕后接收方再发送TCP关闭连接请求，得到发送方确认后，整个TCP连接就关闭了。但此时还需要等待一段时间（2MSL），等这段时间过去后，TCP连接才真正被关闭。

为什么要等待2MSL时间？

  1. 保证客户端发送的最后一个ACK报文段能够到达服务端。
  2. 防止“已失效的连接请求报文段”出现在本连接中。
